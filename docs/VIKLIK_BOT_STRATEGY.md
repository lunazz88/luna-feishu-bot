# Viklik XMP 晨报机器人策略

本文档用于维护正式企业 viklik 中两只飞书机器人的整体处理策略。后续修改机器人逻辑时，先更新本文档，再同步代码，避免规则散落在聊天记录里。

## 目标

用机器人替代数分同学每天重复处理 XMP 广告数据和投手晨报表的流程：

1. robot1 读取每日 XMP 文件和投手晨报表，生成 `日期-ai匹配表`。
2. robot1 同时生成独立的人工核对表。
3. 人工只在核对表里补充或修正必要数据。
4. robot2 读取核对表，把人工补好的数据回写到 `日期-ai匹配表`。

机器人不能修改原始投手晨报表，也不能修改人工版 `日期-XMP` 表。所有自动写入集中在 `日期-ai匹配表` 和独立核对表里。

## 机器人分工

### robot1：xmp晨报数据匹配机器人

负责匹配和生成表：

- 监听 `xmp数据整合分发正式群` 中的标准 XMP 文件。
- 标准文件名格式为 `crawler广告数据报告_YYYY-MM-DD.xlsx`。
- 在操作群中被 `@xmp晨报数据匹配机器人` 并带日期时，读取对应日期的 XMP 文件。
- 找到同日期投手晨报表，例如 `06.25`。
- 基于投手晨报表复制生成 `06.25-ai匹配表`。
- 只写新生成或复用的 `ai匹配表`，不改原始晨报表。
- 生成独立的 `06.25-ai匹配核对表`，供人工处理异常。

robot1 只有被明确 @ 自己时才响应。被 @ 其他机器人时不处理。

### robot2：晨报数据更新机器人

负责回写人工结果：

- 在操作群中被 `@晨报数据更新机器人` 并带日期时执行。
- 读取对应日期的 `日期-ai匹配核对表`。
- 把人工补好的数值回写到 `日期-ai匹配表`。
- 不重新匹配 XMP。
- 不修改原始投手晨报表。
- 不修改人工版 `日期-XMP` 表。
- 不自动改投手字段。

robot2 只有被明确 @ 自己时才响应。被 @ 其他机器人时不处理。

## 触发规则

robot1 示例：

```text
@xmp晨报数据匹配机器人 6.25
@xmp晨报数据匹配机器人 06.25
```

robot2 示例：

```text
@晨报数据更新机器人 6.25
@晨报数据更新机器人 更新6.25数据
```

飞书事件里 @ 可能显示成 `@_user_1`，代码会优先读取事件中的 mentions 元数据判断实际被 @ 的机器人。无法确认被 @ 对象时，不应贸然执行，避免两个机器人串台。

## robot1 匹配策略

### 基础匹配口径

核心匹配字段：

```text
项目名 + code
```

投手用于辅助核对，不作为最终强制匹配条件。国家只作为参考字段，不参与强匹配。

### 数据清洗

通用清洗规则：

- 去除前后空格。
- 空值统一为空字符串。
- 英文字母统一小写比较。
- 项目名去除多余空格。
- 投手名去除多余空格。
- code 统一按字符串处理。

### code 通用规则

以下 code 视为同类候选：

```text
1 / 01 / 001
code1 / code01 / code001
CODE-01 / code_01 / Code 01
tech1 / tech01
```

它们会归一到同一个强 code：`1`。

以下 code 不默认拆数字：

```text
vk01
B4
TK
OH279
```

这些保留业务含义，只做大小写和符号清洗。除非规则表明确映射，不能强行把 `vk01` 当作 `1`、把 `B4` 当作 `4`。

### 唯一命中保护

任何自动写入都必须满足：

- 同一项目内匹配。
- `项目名 + code` 能唯一定位。
- 如果命中多行，不能自动判断时进入人工核对。

这样允许更多通用 code 匹配，同时避免错写。

## robot1 分流表

robot1 生成的独立核对表包括：

### 抓取失败

来源：XMP 中 `抓取状态` 非空的行，例如 `无数据`、`失败`。

用途：人工补抓或补充后续数值。

规则：

- 抓取失败行单独列出。
- 如果能定位到 ai 匹配表行，后续可由 robot2 回填数值。
- 如果数值全空或全 0，robot2 不回填，避免覆盖已有正确数据。

### 投手不一致

来源：`项目名 + code` 能匹配，但 XMP 投手和晨报投手不同。

用途：让人工确认投手归属。

规则：

- 金额为 0 的投手差异不重点干扰人工。
- robot2 不自动修改投手字段。
- 如果人工填了数值，robot2 只回填数值。

### XMP有晨报没有

来源：XMP 有数据，但 ai 匹配表中找不到相同 `项目名 + code` 的行。

用途：人工补充项目名、code、投手名和数值。

规则：

- 如果花费金额为 0，不放入核对表，减少无效人工处理。
- 人工填好项目名、code、投手名和后续数值后，robot2 可补充进 `ai匹配表`。

### 晨报有XMP没有

来源：投手晨报表里有记录，但 XMP 中没有对应 `项目名 + code`。

用途：人工判断是否 XMP 漏抓或无需处理。

规则：

- 如果人工补了数值，robot2 按 `项目名 + code` 回填已有 ai 匹配表行。
- 如果数值全空或全 0，不回填。

## robot2 回写策略

robot2 的原则是：只处理人工明确补充过的数值，不做额外推断。

### 通用保护

以下情况跳过：

- 一行数值字段全空。
- 一行数值字段全为 0。
- 项目名为空。
- code 为空。
- `项目名 + code` 命中多行，无法唯一判断。

数值字段包括：

```text
花费金额
展示次数
点击量
注册人数
首存人数
```

### 抓取失败回写

如果人工在 `抓取失败` 表里补了后续数值：

- 按 `项目名 + code` 找 `ai匹配表` 已有行。
- 唯一命中则回填五个数值字段。
- 不改投手、不改项目、不改 code。

### 投手不一致回写

如果人工在 `投手不一致` 表里补了数值：

- 按 `项目名 + code` 找 `ai匹配表` 已有行。
- 唯一命中则回填五个数值字段。
- 不自动修改投手字段。
- 回复里提示投手需要人工确认。

### XMP有晨报没有处理

如果人工在 `XMP有晨报没有` 表里填好项目名、code、投手名和数值：

- 如果 `ai匹配表` 已有相同 `项目名 + code` 的唯一行，则更新该行数值。
- 如果 `ai匹配表` 没有相同 `项目名 + code`，则新增一行。
- 如果同一次回写中出现重复 `项目名 + code`，只处理一次，避免重复新增。
- 新增行时，如果能在 ai 表已有数据中找到同名投手用户对象，则写入投手；找不到则不写投手，避免 `UserFieldConvFail` 导致整次失败。

### 晨报有XMP没有回写

如果人工在 `晨报有XMP没有` 表里补了数值：

- 按 `项目名 + code` 找 `ai匹配表` 已有行。
- 唯一命中则回填数值。
- 不新增行。

## 金额校验

robot2 回写后会校验：

```text
XMP 原始文件总花费
ai匹配表总花费
差额
```

目标是 `ai匹配表总花费 = XMP 原始文件总花费`。

如果不一致，优先排查：

- 是否有抓取失败行没有人工补数值。
- 是否有 XMP 有但晨报没有的行没有新增。
- 是否有核对表全 0 行错误覆盖了原值。
- 是否使用了不同版本的 XMP 文件。
- 是否飞书 API 网络失败导致部分写入未完成。

## 权限与写入边界

机器人只能写：

- `日期-ai匹配表`
- `日期-ai匹配核对表`

机器人不能写：

- 原始投手晨报表，例如 `06.25`
- 人工版 `日期-XMP`
- XMP 原始 Excel 文件

核对表需要群内成员可编辑，方便人工补充。

## 运行和维护

常用检查：

```bash
npm run check
```

正式 robot1 重启：

```bash
./launch_viklik_xmp_mac.sh
```

正式 robot2 重启：

```bash
./launch_viklik_final_mac.sh
```

常见问题：

- 机器人无反应：检查后台日志是否 `ws client ready`。
- API 失败：常见为 `Client network socket disconnected before secure TLS connection was established`，属于飞书 API/TLS 网络连接问题。
- 投手字段写入失败：通常是飞书人员字段不能直接写文本，新增行时应尽量复用已有用户对象。
- 两个机器人串台：检查是否只响应被 @ 的目标机器人。

## 维护原则

1. 新规则先写入本文档，再改代码。
2. 不把密钥、`.env`、本地输出文件提交到 Git。
3. 不把临时分析脚本混进生产提交，除非明确要长期维护。
4. 匹配规则优先通用化，避免为单个项目无限打补丁。
5. 自动写入必须有唯一命中保护，宁可进入人工核对，也不要写错。
